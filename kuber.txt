# 06 — Calico CNI

**Предварительные условия:**

---

## 6.1. Подготовка CNI директорий

На **всех 5 нодах**:

```bash
# Очистка предыдущих CNI конфигов
rm -rf /etc/cni/net.d/* /opt/cni/bin/*
mkdir -p /etc/cni/net.d /opt/cni/bin
```

---

## 6.2. Применение Calico CRDs

> **⚠️ Ошибка #14 (из истории):** Calico v3.30.7 содержит CRDs встроенными в calico.yaml. Отдельный файл CRDs не требуется, но если вы хотите применить их отдельно — извлеките из официального calico.yaml.

```bash
# Применение CRDs (если отдельным файлом)
kubectl apply -f /root/projects/k8s_mm_local_test/manifests/calico-crd.yaml

# Или применить всё сразу
kubectl apply -f /root/projects/k8s_mm_local_test/manifests/calico.yaml
```

---

## 6.3. Ключевые параметры Calico manifests

Файл `/root/projects/k8s_mm_local_test/manifests/calico.yaml` содержит следующие критичные настройки:

### 6.3.1. Backend: VXLAN CrossSubnet

```yaml
# calico-config ConfigMap
CALICO_IPV4POOL_IPIP: Never
CALICO_IPV4POOL_VXLAN: CrossSubnet
```

> **Почему VXLAN CrossSubnet:** BGP требует маршрутизатор и дополнительную настройку. VXLAN CrossSubnet работает «из коробки» в изолированной среде. IPIP отключён.

### 6.3.2. Interface detection

```yaml
IP_AUTODETECTION_METHOD: "interface=enp1s0"
```

> **⚠️ Ошибка (из TROUBLESHOOTING.md):** При восстановлении из бэкапа Calico мог конфликтовать с kube-ipvs0. `interface=enp1s0` гарантирует правильный выбор интерфейса.

### 6.3.3. MTU

```yaml
FELIX_VXLANMTU: "1450"
```

MTU 1450 = 1500 (стандартный) - 50 (VXLAN overhead).

### 6.3.4. RBAC

Calico требует дополнительные RBAC правила:

```yaml
# ClusterRole calico-node — должен включать:
- apiGroups: ['']
  resourceNames: [calico-cni-plugin]
  resources: ['serviceaccounts/token']
  verbs: [create]

# ClusterRoleBinding calico-cni-plugin:
subjects:
  - kind: ServiceAccount
    name: calico-cni-plugin
    namespace: kube-system  # НЕ calico-system!

# ClusterRoleBinding calico-tier-getter:
subjects:
  - apiGroup: rbac.authorization.k8s.io
    kind: User
    name: system:kube-controller-manager
```

> **⚠️ Ошибка #9 (из истории):** Без ClusterRole для `serviceaccounts/token` Calico не может создать ServiceAccount token → pod'ы не запускаются.

### 6.3.5. Health probes

```yaml
# liveness probe (НЕ httpGet :9099)
livenessProbe:
  exec:
    command: ["/bin/calico-node", "-felix-live", "-bird-live"]

# readiness probe (НЕ httpGet :9099)
readinessProbe:
  exec:
    command: ["/bin/calico-node", "-felix-ready", "-bird-ready"]
```

> **⚠️ Ошибки #10, #11 (из истории):** httpGet на порт 9099 не работает в некоторых версиях Calico. Используйте `exec` команды.

### 6.3.6. Install CNI init-container

```yaml
# initContainer install-cni
env:
  - name: CNI_CONF_NAME
    value: 10-calico.conflist
```

> **⚠️ Ошибка (из истории):** init-контейнер `install-cni` может падать с `FATAL: no such file or directory` после успешного создания CNI конфига. CNI конфиг и бинарники установлены корректно, но init-контейнер остаётся в CrashLoopBackOff. Это не влияет на работу сети — pod'ы получают IP адреса. Для исправления можно добавить `CNI_CONF_NAME` env var или обновить версию Calico.

## 6.6. Нагрузка на etcd — предупреждение

> **⚠️ Критическая проблема (из истории):** При первоначальной установке Calico на обычные диски (не NVMe) etcd был перегружен — latency 3-4 секунды, кластер деградировал.
>
> **Решение:** Диски control-plane нод должны быть на NVMe pool (`/opt/libvirtd/`). См. [`01-ИНФРАСТРУКТУРА.md`](01-ИНФРАСТРУКТУРА.md), раздел 1.1.
>
> **Признаки перегрузки etcd:**
> - `apply request took too long` в логах etcd
> - `slow fdatasync` > 1 секунды
> - kube-apiserver перезапускается
> - kube-vip теряет лидерство

## 7.1. Helm установка cert-manager

```bash
helm install cert-manager \
  /root/projects/k8s_mm_local_test/helm/cert-manager/cert-manager-v1.19.4.tgz \
  --create-namespace \
  --namespace cert-manager \
  --set installCRDs=true \
  --set image.repository=harbor.local.test:8443/library/quay.io/jetstack/cert-manager-controller \
  --set image.tag=v1.19.4 \
  --set cainjector.image.repository=harbor.local.test:8443/library/quay.io/jetstack/cert-manager-cainjector \
  --set cainjector.image.tag=v1.19.4 \
  --set webhook.image.repository=harbor.local.test:8443/library/quay.io/jetstack/cert-manager-webhook \
  --set webhook.image.tag=v1.19.4 \
  --set startupapicheck.image.repository=harbor.local.test:8443/library/quay.io/jetstack/cert-manager-startupapicheck \
  --set startupapicheck.image.tag=v1.19.4
```

> **⚠️ Ошибка #12 (из истории):** startupapicheck использует образ `cert-manager-startupapicheck`, НЕ `cert-manager-ctl`. Начиная с v1.15.0 образ переименован.

## 7.2. Vault Token Secret

```bash
kubectl create secret generic vault-token \
  -n cert-manager \
  --from-literal=token=<VAULT_CERT_TOKEN>
```

## 7.3. Vault ClusterIssuer

```yaml
# cert-manager-vault-issuer.yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: vault-issuer
spec:
  vault:
    server: https://192.168.122.2:8200
    path: pki_int/sign/k8s-api
    caBundle: <BASE64_VAULT_TLS_CERT>
    auth:
      tokenSecretRef:
        name: vault-token
        key: token
```

```bash
kubectl apply -f /root/projects/k8s_mm_local_test/manifests/cert-manager-vault-issuer.yaml
```

> **⚠️ Ошибка #13 (из истории):** caBundle = TLS-сертификат Vault (`vault.crt`), НЕ PKI CA-сертификат. cert-manager использует его для TLS-соединения с Vault, не для верификации PKI-цепочки.

---

## 7.6. ⚠️ Vault sealed — что происходит

> **⚠️ Ошибка #1 (из истории):** Если Vault запечатан после перезапуска:
> - cert-manager **не отзывает** существующие сертификаты
> - Существующие сертификаты продолжают работать до истечения срока
> - Перевыпуск невозможен до unseal
>
> **Решение:** Unseal Vault + force-update clusterissuer:
> ```bash
> kubectl patch clusterissuer vault-issuer --type merge \
>   -p '{"metadata":{"annotations":{"cert-manager.io/force-update":"'"$(date +%s)"'"}}}'
> ```

# 08 — Istio Service Mesh

**Цель:** Установить Istio service mesh с mTLS, CNI, и базовой безопасностью.

## 8.0. Критически важное: ENABLE_NATIVE_SIDECARS

> **⚠️ КРИТИЧНО:** Для Istio 1.29.1 + Kubernetes 1.35 необходимо установить `ENABLE_NATIVE_SIDECARS=false`. Без этого sidecar инжектируется как init container, что ломает маршрутизацию трафика (503 cluster_not_found).

---

## 8.1. Istio CA Secret

Создайте namespace и Secret с 4 файлами Istio CA:

```bash
kubectl create namespace istio-system

kubectl create secret generic cacerts -n istio-system \
  --from-file=ca-cert.pem=/tmp/istio-ca/ca-cert.pem \
  --from-file=ca-key.pem=/tmp/istio-ca/ca-key.pem \
  --from-file=root-cert.pem=/tmp/istio-ca/root-cert.pem \
  --from-file=cert-chain.pem=/tmp/istio-ca/cert-chain.pem
```

> **⚠️ Ошибка #5 (из истории):** Istio CA НЕЛЬЗЯ выпустить через cert-manager + Vault PKI. Vault не поддерживает `is_ca=true`. Istio CA генерируется локально через openssl и подписывается Vault Root CA.

## 11.2. Конфигурация

```yaml
# metallb-config.yaml
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  name: ingress-pool
  namespace: metallb-system
spec:
  addresses:
    - 192.168.122.200-250
---
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  name: egress-pool
  namespace: metallb-system
spec:
  addresses:
    - 192.168.130.2-250
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement
metadata:
  name: ingress-l2
  namespace: metallb-system
spec:
  ipAddressPools:
    - ingress-pool
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement
metadata:
  name: egress-l2
  namespace: metallb-system
spec:
  ipAddressPools:
    - egress-pool
```

> **⚠️ Ошибка #3 (из TROUBLESHOOTING.md):** НЕ указывайте `interfaces: ["eth0"]` в L2Advertisement. На RED OS 8.0 интерфейс называется `enp1s0`. MetalLB автоматически определит правильный интерфейс по IP адресу.
>
> **⚠️ Ошибка #11 (из ОШИБКИ-НОВОЕ-РАЗВЁРТЫВАНИЕ.md):** Убраны `interfaces` из L2Advertisement — MetalLB автоопределяет интерфейс по IP.

### 11.2.1. Дополнительные IP на worker'ах

Для пула `192.168.130.0/24` (egress) на worker-нодах нужен дополнительный IP:

```bash
# k8s-w1
nmcli connection modify enp1s0 +ipv4.addresses '192.168.130.251/24'
nmcli connection up enp1s0

# k8s-w2
nmcli connection modify enp1s0 +ipv4.addresses '192.168.130.252/24'
nmcli connection up enp1s0
```

> **Примечание:** Без дополнительного IP MetalLB не сможет анонсировать IP из пула `192.168.130.0/24` через ARP.

---

## 11.3. Speaker только на worker'ах

> **Рекомендация:** MetalLB speaker должен работать только на worker-нодах для безопасности и сохранения source IP клиента.

```bash
# Обновить DaemonSet speaker для работы только на worker'ах
kubectl patch daemonset speaker -n metallb-system --type='json' \
  -p='[{"op":"add","path":"/spec/template/spec/affinity","value":{"nodeAffinity":{"requiredDuringSchedulingIgnoredDuringExecution":{"nodeSelectorTerms":[{"matchExpressions":[{"key":"node-role.kubernetes.io/control-plane","operator":"DoesNotExist"}]}]}}}}]'
```

---

## 11.4. ⚠️ Конфликт kube-vip + MetalLB

> **⚠️ Ошибка #4 (критическая, из TROUBLESHOOTING.md):** kube-vip с `svc_enable: "true"` конфликтует с MetalLB за LoadBalancer IP.
>
> **Симптомы:**
> - ARP указывает на master-ноду, а не worker
> - `curl http://192.168.122.200:80/` → timeout
> - На мастерах: `ip addr show enp1s0 | grep deprecated` → LoadBalancer IP /32
>
> **Решение:** `svc_enable: "false"` в kube-vip manifest на всех control-plane нодах. Удалить deprecated IP: `ip addr flush dev enp1s0 scope global`.
>
> **Разделение ответственности:**
> - kube-vip — только для control-plane VIP (192.168.122.40)
> - MetalLB — для всех LoadBalancer сервисов
