## 🔹 1. Что такое ArgoCD и ключевые принципы
- **GitOps**: Кластер должен соответствовать тому, что закоммичено в Git. ArgoCD постоянно сравнивает *желаемое состояние* (Git) с *фактическим* (Kubernetes) и синхронизирует их.
- **Декларативный подход**: Все приложения описываются манифестами Kubernetes (`Application`), которые тоже хранятся в Git.
- **Поддерживаемые форматы**: Plain YAML, Helm, Kustomize, Jsonnet, ksonnet (устарел).
- **Синхронизация**: Ручная или автоматическая. ArgoCD умеет создавать, обновлять и удалять (`prune`) ресурсы.

## 🔹 2. Быстрая установка в Kubernetes
```bash
# 1. Создаём namespace
kubectl create namespace argocd

# 2. Устанавливаем ArgoCD (официальные манифесты)
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# 3. Ждём готовности
kubectl wait --for=condition=Ready pods -n argocd -l app.kubernetes.io/name=argocd-server --timeout=120s
```

### 🔐 Первый вход
```bash
# Получить пароль администратора
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

# Открыть UI через port-forward
kubectl port-forward svc/argocd-server -n argocd 8080:443
# Браузер: https://localhost:8080 (логин: admin, пароль выше)
```

### 💻 CLI
```bash
# Linux/macOS
curl -sSL -o argocd https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
chmod +x argocd && sudo mv argocd /usr/local/bin/

# Логин
argocd login localhost:8080 --insecure
```

## 🔹 3. Создание первого приложения
Рекомендуется описывать приложения через манифесты, а не через UI/CLI.

### 📄 `my-app.yaml`
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: demo-app
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io # удаляет ресурсы при удалении Application
spec:
  project: default
  source:
    repoURL: https://github.com/argoproj/argocd-example-apps.git
    targetRevision: HEAD
    path: guestbook
  destination:
    server: https://kubernetes.default.svc
    namespace: demo-ns
  syncPolicy:
    automated:
      prune: true      # удалять ресурсы, которых нет в Git
      selfHeal: true   # автоматически исправлять дрейф конфигурации
    syncOptions:
      - CreateNamespace=true
```

Применяем:
```bash
kubectl apply -f my-app.yaml
```

ArgoCD автоматически создаст namespace `demo-ns` (благодаря `CreateNamespace=true`) и задеплоит приложение.

---
## 🔹 4. Ключевые параметры `syncPolicy`
| Параметр | Описание |
|----------|----------|
| `automated.prune: true` | Удаляет ресурсы, которые есть в кластере, но отсутствуют в Git |
| `automated.selfHeal: true` | Если кто-то вручную изменил ресурс в K8s, ArgoCD вернёт его к Git-состоянию |
| `syncOptions` | `CreateNamespace=true`, `ApplyOutOfSyncOnly=true`, `Replace=true`, `ServerSideApply=true` |
| `retry` | Настройка повторов при падении синхронизации |
| `ignoreDifferences` | Игнорировать дрейф в конкретных полях (полезно для managed-полей операторов) |

## 🔹 5. AppProject: изоляция и безопасность
`default` проект подходит для тестов. Для продакшена создавайте отдельные проекты:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: backend-team
  namespace: argocd
spec:
  description: Backend team apps
  sourceRepos:
    - https://github.com/your-org/backend-k8s.git
  destinations:
    - namespace: backend-*
      server: https://kubernetes.default.svc
  clusterResourceWhitelist:
    - group: ''
      kind: Namespace
```
Проекты позволяют ограничивать:
- Репозитории
- Целевые кластеры/неймспейсы
- Доступные API-группы
- RBAC-роли для команд

## 🔹 6. Полезные команды CLI
```bash
argocd app list                     # список приложений
argocd app get demo-app             # статус и детали
argocd app sync demo-app            # ручная синхронизация
argocd app history demo-app         # история синхронизаций
argocd app logs demo-app            # логи синхронизации
argocd app diff demo-app            # показать дрейф до sync
argocd app set demo-app --sync-policy automated --auto-prune --self-heal
```

## 🔹 7. Production-рекомендации
1. **Храните ArgoCD в Git**  
   Используйте паттерн **App of Apps**: один корневой `Application` деплоит остальные приложения из репозитория.
2. **Не включайте `auto-sync` в prod без осторожности**  
   Начинайте с ручного sync или `selfHeal: true` + `prune: false`. Включайте `prune` только после тестов.
3. **Настройте вебхуки**  
   Git-хуки (GitHub/GitLab) мгновенно уведомляют ArgoCD о новых коммитах, убирая polling.
4. **Управляйте секретами отдельно**  
   ArgoCD не шифрует секреты по умолчанию. Используйте Sealed Secrets, External Secrets, SOPS или Vault.
5. **Версионируйте `targetRevision`**  
   В prod используйте теги/хэши (`v1.2.3` или `abc123f`), а не `main`/`HEAD`.
6. **Мониторинг**  
   ArgoCD экспортирует метрики Prometheus. Графана-дашборды уже есть в `manifests/monitoring/grafana/`.

## 🔹 8. Отладка и типичные проблемы
| Симптом | Причина | Решение |
|--------|---------|---------|
| `Synced` но `Healthy: False` | Ресурсы не прошли health-check | Проверьте readiness/liveness, события в K8s, кастомные health-скрипты |
| `ComparisonError` | Нет доступа к репо или неверный путь | Проверьте токен, SSH-ключ, `path`, `targetRevision` |
| `SyncFailed` | Права RBAC, конфликт ресурсов | Проверьте ServiceAccount ArgoCD, `clusterrole`/`rolebinding` |
| Ресурсы не удаляются при `prune: true` | Не добавлен finalizer или включена аннотация `argocd.argoproj.io/sync-wave` | Проверьте аннотации, используйте `--prune` вручную для отладки |

### 3.1. cert-manager + Vault Issuer
**`values/cert-manager-values.yaml`**
```yaml
image:
  repository: harbor.local.test:8443/library/quay.io/jetstack/cert-manager-controller
  tag: v1.19.4
cainjector:
  image:
    repository: harbor.local.test:8443/library/quay.io/jetstack/cert-manager-cainjector
    tag: v1.19.4
webhook:
  image:
    repository: harbor.local.test:8443/library/quay.io/jetstack/cert-manager-webhook
    tag: v1.19.4
startupapicheck:
  image:
    repository: harbor.local.test:8443/library/quay.io/jetstack/cert-manager-startupapicheck
    tag: v1.19.4
installCRDs: true
```
**`apps/cert-manager.yaml`**
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: cert-manager
  namespace: argocd
spec:
  project: infra-stack
  source:
    repoURL: https://charts.jetstack.io
    targetRevision: v1.19.4
    chart: cert-manager
    helm:
      releaseName: cert-manager
      valueFiles:
        - ../../values/cert-manager-values.yaml
  destination:
    server: https://kubernetes.default.svc
    namespace: cert-manager
  syncPolicy:
    automated: { prune: true, selfHeal: true }
    syncOptions:
      - CreateNamespace=true
      - ServerSideApply=true
```
*Секрет Vault-токена*: Не храните в Git. Используйте **External Secrets Operator** или **SOPS**. Если временно нужно вручную:
```yaml
# manifests/vault-token.yaml (управляется ArgoCD)
apiVersion: v1
kind: Secret
metadata:
  name: vault-token
  namespace: cert-manager
type: Opaque
data:
  token: <BASE64>
```

## 🔐 4. Управление секретами в GitOps
В вашей инструкции используется `kubectl create secret generic vault-token`. В ArgoCD **не коммитьте секреты в plain text**. Выберите один из вариантов:
1. **Sealed Secrets / SOPS**: Шифруйте YAML в репозитории, ArgoCD расшифрует на лету.
2. **External Secrets Operator (ESO)**: Арго деплоит `ExternalSecret`, который забирает токен из Vault по API. Это наиболее безопасно для `cert-manager-vault-issuer`.
3. **ArgoCD Vault Plugin**: Подключается напрямую к Vault при синхронизации.

## ⚠️ 6. Архитектурные нюансы (важно!)
1. **Calico CNI**: Сетевые плагины обычно устанавливаются **до** ArgoCD, т.к. ArgoCD-сервер и agents работают как pod'ы и требуют сети. Если вы всё же хотите управлять Calico через ArgoCD, используйте `App of Apps` pattern и убедитесь, что `calico-node` DaemonSet не зависит от ArgoCD-сервера.
2. **kube-vip vs MetalLB**: В вашей инструкции уже отмечен конфликт. Убедитесь, что `kube-vip` манифест на control-plane содержит `svc_enable: "false"`. Этот манифест лучше вынести из ArgoCD и управлять им через Ansible/Terraform на этапе bootstrap кластера.
3. **Порядок синхронизации**: Используйте `sync-wave` для зависимостей:
   ```yaml
   metadata:
     annotations:
       argocd.argoproj.io/sync-wave: "1" # cert-manager, CRDs
   ```
   `istio-base` → `cert-manager` → `istiod` → `monitoring` → `gateway-api`

## 🔒 Принципы GitOps в air-gapped среде
1. **Никаких внешних ссылок**. ArgoCD должен работать только с внутренним Git-репозиторием и внутренним Harbor.
2. **Helm-чарты хранятся в Git**. Распакованные `.tgz` из вашей инструкции коммитятся как директории с `Chart.yaml`.
3. **Все образы** уже указаны в `values.yaml` через `harbor.local.test:8443/...` (это учтено корректно).
4. **CRD и plain YAML** применяются через `source.path` с приоритетом синхронизации (`sync-wave`).

## 🛠️ Исправленные `Application` манифесты (только внутренние источники)

### 1. Calico (plain YAML)
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: calico
  namespace: argocd
  annotations:
    argocd.argoproj.io/sync-wave: "-2" # Первым
spec:
  project: infra-stack
  source:
    repoURL: https://git.internal/k8s/gitops-repo.git
    targetRevision: main
    path: manifests/calico
  destination:
    server: https://kubernetes.default.svc
    namespace: kube-system
  syncPolicy:
    automated: { prune: false, selfHeal: true } # CNI не удаляем автоматически!
```

## ⚙️ Настройка ArgoCD для работы с внутренним Harbor
ArgoCD по умолчанию не умеет авторизовываться в приватных реестрах без явной настройки.

### 1. Добавить креды Harbor в ArgoCD
```bash
argocd repo add harbor.local.test:8443 \
  --type helm \
  --username <harbor-user> \
  --password <harbor-pass> \
  --insecure-skip-server-verification # если самоподписанный TLS
```
Или через UI: `Settings → Repositories → Connect Repo`.

### 2. Если используете Helm OCI Registry (Harbor 2.5+)
Вместо распаковки чартов в Git можно загрузить их в Harbor:
```bash
helm package cert-manager-v1.19.4.tgz
helm push cert-manager-v1.19.4.tgz oci://harbor.local.test:8443/library/charts
```
Тогда `Application` будет выглядеть так:
```yaml
source:
  repoURL: oci://harbor.local.test:8443/library/charts/cert-manager
  targetRevision: v1.19.4
  helm: { releaseName: cert-manager }
```
Это часто удобнее для offline-кластеров.

## 📋 Чек-лист полной адаптации под air-gapped
| Шаг | Статус | Комментарий |
|-----|--------|-------------|
| ✅ | Все образы выкачаны и залиты в `harbor.local.test:8443` | Проверяется через `crane ls` или Harbor UI |
| ✅ | Helm `.tgz` распакованы в `charts/` и закоммичены в Git | Или загружены в Harbor как OCI |
| ✅ | `repoURL` в `Application` указывают только на внутренний Git/Harbor | Нет `jetstack.io`, `istio.io`, `metallb.github.io` |
| ✅ | В `values.yaml` полные пути к образам `harbor.local.test:8443/...` | Проверить отсутствие `docker.io`, `gcr.io`, `quay.io` |
| ✅ | Calico CNI имеет `prune: false` | Защита от случайного удаления сети |
| ✅ | CRD применяются с `sync-wave: "-2"` или отдельным Application | Гарантия порядка: CRD → Controllers → Apps |
| ✅ | Настроен `imagePullSecret` в `argocd-server` и `argocd-application-controller` | Если Harbor требует auth для pull |
| ✅ | Отключены внешние вебхуки/телеметрия | `metricsBackends=prometheus` (локально), нет `postInstall` с `curlimages` |

## 💡 Рекомендация по bootstrap в offline
1. Разверните ArgoCD вручную через `kubectl apply -f argocd-install.yaml` (предварительно заменив все `image:` на `harbor.local.test:8443/...`).
2. Примените `AppProject` и базовые `Application` с `syncPolicy.automated: false`.
3. Проверьте дрейф: `argocd app diff calico`, `argocd app diff cert-manager`.
4. После валидации включите `automated: { selfHeal: true }` по одному компоненту.

Если нужно, могу сгенерировать:
- `argocd-cm` конфиг с предустановленными репозиториями и TLS-сертификатами Harbor
- Kustomize-патчи для тюнинга VictoriaLogs/Calico без `kubectl patch`
- Скрипт для массовой распаковки `.tgz` из инструкции в Git-структуру

Уточните, какой компонент требует детальной адаптации в первую очередь.
